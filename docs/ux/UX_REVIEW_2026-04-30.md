## UX/UI status summary

Current UX status: **NEEDS WORK**.

The project has a premium dark TV visual direction, remote input foundation and focus system, but there is a critical mismatch between the documented menu source of truth and the runtime UI menu definition.

## Critical finding 1 — menu source mismatch

### Current UX status

**NEEDS WORK / Critical**

### What was found

`docs/product/VISUAL_REFERENCE.md` states that the left menu content must be preserved from:

`android/app/src/main/assets/web/menu.txt`

`menu.txt` currently defines the full menu:

```text
Главная
Каталог
Поиск
AI подбор
Детали
Player Pro
Библиотека
Серии
Источники
Torrent
QR импорт
Диагностика
Обновления
Настройки
```

However, `android/app/src/main/assets/web/ui.js` currently hardcodes a shorter menu:

```js
['Главная','Поиск','Каталог','Библиотека','Источники','Настройки']
```

### UX impact

- The actual left menu does not match the documented visual/reference source.
- Important flows are not directly available from the left navigation:
  - AI подбор
  - Детали
  - Player Pro
  - Серии
  - Torrent
  - QR импорт
  - Диагностика
  - Обновления
- The user journey described in `VISUAL_REFERENCE.md` cannot be fully executed from the menu.
- Engineering chats may incorrectly change or simplify the menu because code and docs disagree.

### Recommended change

Do **not** manually redesign the left menu content.

Instead, make `ui.js` load or mirror the menu from `android/app/src/main/assets/web/menu.txt`, so the runtime menu uses the documented source of truth.

Safe implementation options:

1. Preferred: load `menu.txt` at startup and fall back to embedded default only if loading fails.
2. Acceptable: update the `AsUI.menu` array to exactly match `menu.txt` and add a code comment saying `menu.txt` is the source of truth.

### Priority

**Critical**

## Finding 3 — input/focus foundation is strong but needs real TV QA

### Current UX status

**GOOD foundation / QA REQUIRED**

### What was found

`input.js` implements:

- visible focusable item collection;
- directional D-pad movement by geometric scoring;
- Enter activation;
- Back/Escape routing;
- scroll focused element into view;
- filtering invisible/disabled elements.

### UX impact

This is a good foundation for Android TV UX, but it still needs emulator/device validation because TV remote behavior can differ from browser keyboard behavior.

### Recommended change

Run a real D-pad audit on Android TV emulator or Mi Box S:

- left/right movement through shelves;
- up/down movement between hero, shelves and menu;
- focus recovery after screen change;
- Back behavior from nested screens;
- no focus traps in inputs/textareas.

### Priority

**Critical QA**

### Related backlog IDs

- `ASG-002`
- `ASG-092`

### Files likely affected

- `android/app/src/main/assets/web/input.js`
- `android/app/src/main/assets/web/ui.js`
- `android/app/src/main/assets/web/main.js`
